We had 18 million rows.
Postgres was slowing down search.

The question was simple:
do we add a GIN index or cache the results in Redis?

GIN indexes in PostgreSQL are built for searching inside data.
Think tags, text, JSON fields, and “find me anything containing this.”

Redis is an in-memory cache.
Think “keep the answer on a sticky note” for fast repeat reads.

We tested both on the same endpoint.

GIN got the query from 420 ms to 38 ms.
Redis got repeated reads to 4 ms.
But Redis also added a second system, cache invalidation, and 2 days of edge cases.

GIN took one migration.
No new service.
No stale data.
No extra code path to debug at 2 a.m.

We picked GIN.

The lesson: if Postgres can answer the question well enough, keep the work inside Postgres.
A second system should earn its place, not just look faster in a benchmark.
