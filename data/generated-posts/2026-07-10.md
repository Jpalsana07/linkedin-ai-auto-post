# LinkedIn Post

Topic: Next.js in Production

Date: 2026-07-10

---

Next.js looked fine in staging.

We were building a customer portal for 4,000 paid users, and the first real traffic wave made pages stall for 11 seconds.

The homepage loaded.
Then the dashboard went blank.

We had shipped it on Next.js 14, using server components (rendered on the server) and an API route (a backend endpoint) for every widget.

On my laptop, it felt instant.

In production, one slow database query held the whole page open.
The server was waiting to finish everything before sending HTML, so the browser had nothing to show.

We tried bigger instances first.
Wrong fix.

The real problem was that we had mixed “nice to have” data with “must show now” data in the same request.
One stock chart and one notification count were blocking the whole customer page.

We split the page.
The shell rendered first.
The expensive parts loaded after, with Suspense (a loading boundary) and cached data in Redis (a fast memory store).

Latency dropped from 11 seconds to 1.8.
Support tickets fell the same week.

The lesson: in production, speed is not about making everything faster.
It’s about deciding what can wait.

What do you think should render first when a page is under pressure?
