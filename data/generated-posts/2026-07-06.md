# LinkedIn Post

Topic: Agent Reliability

Date: 2026-07-06

---

We let the agent send money alone.

It was a support workflow for refunds.
The stakes were real: if it worked, we could cut response time from hours to minutes.

For 3 days, it looked fine.
Then a customer got the same refund twice.

The bug was not the model.
It was us assuming one good answer meant one safe action.

The agent used OpenAI for reasoning and Stripe for payments.
Stripe is the payment processor.

We had a tool schema — the menu of actions the bot can take — and it had only one rule: “issue refund.”
No idempotency key, which is a unique stamp that prevents duplicate actions.
No human check on money-moving steps.

So when the request retried after a timeout, the second run looked identical.
The bot was “correct.”
The system was not.

We added a simple gate after that:
the agent could draft the refund,
but a person had to approve the final charge.

That single change cut duplicate refunds to zero.

The lesson: an agent is not reliable just because each step sounds reasonable.
It has to be safe when the same step happens twice.

Where have you seen a system look correct on the first pass and still fail in the real world?
