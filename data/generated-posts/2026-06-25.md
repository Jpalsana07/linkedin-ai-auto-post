TypeScript made two lies

We were building a checkout flow, and the bug was simple: cart data said one thing, the API expected another.

Two tools could have caught it before prod: TypeScript and Zod, the runtime shape checker.

TypeScript is great at compile time, which means it helps while you’re writing code.

Zod checks at runtime, which means it verifies the data after it arrives from an API, form, or webhook.

We had 18 files with “safe” types that looked perfect in the editor.

Then a Stripe webhook sent a field as null instead of string, and the app crashed anyway.

That’s the pitfall: TypeScript can only protect the code it sees.

It cannot trust the outside world.

Zod slowed us down by maybe 30 minutes to add schemas, but it saved a full day of debugging the next time a payload drifted.

TypeScript was still essential.

But for anything crossing a network boundary, Zod won.

My lesson: static types are for your code, runtime validation is for reality.

Which side have you trusted too much, the editor or the data?
