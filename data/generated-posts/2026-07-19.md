We picked the wrong Postgres tool.

We needed faster reads on a product page.
Not a new database.
Just a way to answer the same question 10,000 times without making Postgres sweat.

The two candidates were Redis and pgvector.

Redis is a memory store, so it keeps hot data in RAM for very fast access.
pgvector is a PostgreSQL extension for similarity search, which means finding “close enough” items by comparing embeddings, or number lists that represent meaning.

Redis was faster on paper.
Single-digit millisecond reads.
But it meant another service, another cache to warm, and another place for stale data to hide.

pgvector was slower.
Our queries landed around 40 ms.
But it lived next to the data, used one backup system, and shipped in 2 days instead of 2 weeks.

That mattered more than speed.

The lesson: choose the tool that removes moving parts first.
Performance is nice.
Operational simplicity keeps the system alive when the page gets hammered at 9 a.m.

We kept pgvector and deleted the Redis experiment.

What’s a case where the slower Postgres-native option beat the “faster” tool for you?
